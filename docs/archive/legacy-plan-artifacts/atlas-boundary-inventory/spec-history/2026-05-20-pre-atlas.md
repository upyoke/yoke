# Spec: Gen 3 destination map: classify every Yoke surface and config setting for the cloud cutover

## Operator Intent To Preserve

The original operator intent still stands:

> catalogue every python function in the repo that agents could or would potentially call directly
>
> block agents from doing anything not through the api completely
>
> carve out a limited list of specific sanctioned exceptions or exception classes, with justifications for each
>
> identify gaps that need to become first class apis

The strategic correction is **scope and framing**. This ticket is not a small enforcement-program kickoff. It is the **foundational Gen 3 design analysis** — the lynchpin that determines what every Yoke surface AND every config setting becomes in cloud Yoke. Every downstream Gen 3 ticket (conversion slices, install bundle, env binding, snapshot capability, ephemeral envs, Yoke-the-project cutover) consumes this ticket's hand-authored design decisions. Doing the design once here vs scattering it across 6 follow-up tickets is dramatically cheaper.

## Why This Exists

Gen 3 closes by making Yoke a deployed service that operates Buzz and Yoke-the-project from project-local installs (per-env bundle + credential, no local Yoke code in customer repos). For that to be real, every Yoke-owned entrance an agent currently reaches AND every config setting agents or skills currently depend on must have a defined Gen 3 destination: become a registered HTTP function call, become an agent-side harness recipe, become a static install artifact, move to per-env runtime state, move to per-project git-tracked state, or be deleted.

This ticket produces the foundational destination map. It is the input to:

- Phase 0 Stage-2 follow-up conversion slices (operationalize the API decisions)
- Phase 1 cloud-readiness reshape (consumes the config classification to know what schema additions land)
- Phase 2 cloud infrastructure (snapshot/ephemeral/break-glass primitives know what to snapshot, what to template, what to migrate)
- Phase 3 install bundle and env-binding (skills/packets/configs rendered per-env from this map; `yoke switch` consumes this map to know what to swap)
- Phase 4 Yoke-the-project cutover (migration tool consumes the map to know what data moves where)

Two recent landed facts shape this version:

- YOK-1665 (Gen 2) landed the function-call dispatcher and registry. The live registry has registered ids; the inventory runner calls handler registration before reporting.
- Subsequent Gen 2 landings (YOK-1688, YOK-1695, YOK-1696, YOK-1700, YOK-1701, YOK-1706, YOK-1707, YOK-1711, YOK-1713, YOK-1714 through YOK-1720, YOK-1729, YOK-1730) added lint, packet, ownership, session, actor, watcher, and advance-orchestration facts the inventory must consume. The stale `harness_sessions.execution_scope.*` family is removed because YOK-1691 deleted that surface.

The old 2026-05-13 deep audit classified 334,925 rows and remained the reference evidence — but its generator was artifact-local, ignored, and hardcoded to one machine/path. This ticket makes it reproducible from repo-owned code, then expands the classification to cover configs as well.

This ticket produces the durable Gen 3 destination map. It does **not** implement every road on the map.

## Scope: two parallel inventories sharing one classification spine

**Surface inventory.** Every Yoke-owned entrance an agent could reach — Python recipes (`python3 -m runtime.api...`), CLI adapters, registered function ids, `db_router` subcommands, `service_client` surfaces, direct module imports, HTTP routes (where they exist today), hook event handlers.

**Config inventory.** Every Yoke-owned setting that affects runtime behavior — `data/config` keys (e.g. `lint_polling_mode`, `session_reactivation_reacquire_window_s`), `project_capabilities` config blobs, hook config defaults, lint-mode defaults, harness manifest entries, schema-derived defaults, env-var-based fallbacks.

The two inventories share family identity where natural — `lint_polling_mode` config and the `Bash` PreToolUse lint surface both belong to the `lint` family. Family-level decisions cover both halves.

**The inventory includes both surfaces that exist today AND surfaces that don't yet exist but Gen 3 requires.** Examples of future surfaces: the server-side `/hooks/evaluate` dispatcher endpoint, the per-event server-side policy functions cloud Yoke's curl-based hooks dispatch into. Each future surface gets its own row with its own disposition exactly like an existing surface. The runner outputs only what exists today; per-family design pages declare additional planned rows the conversion slices will create.

## Per-Surface Classification

Applied at the **surface family** level (~18 families: the 17 that emerged from the 2026-05-13 deep audit — `items`, `claims`, `lifecycle`, `harness_sessions`, `merge_deploy`, `workflow_item`, `qa`, `doctor`, `dependency_graph`, `github_sync`, `project_structure`, `events`, `context_packets`, `function_inventory`, `control_plane_policy`, `api_namespace_cleanup`, `api_gap_or_stale_teaching` — plus `hooks`, which carries two row entries: `hooks.install_config` for the curl-line install artifact in `.claude/settings.json` / `.codex/hooks.json`, and `hooks.policy_logic` for the server-side `/hooks/evaluate` dispatcher and per-event policy functions cloud Yoke runs internally). Individual rows inherit their family's classification; the family-level decisions drive follow-up work.

Each surface family carries six columns. The disposition column tells the migration what to do; the five signals are informational and orthogonal.

### Disposition (one of six — what the migration does to this surface)

- `wrap_as_function` — becomes a registered first-class function id agents call.
- `agent_executes_via_harness` — agent runs it using native harness tools (Bash/Read/Grep/Monitor/etc.) and reports results via a function call. Not a Yoke-side surface; a teaching pattern in skill markdown.
- `static_install_artifact` — bundled as install file (packet, skill markdown, hook config template). Served by Yoke core; downloaded on `yoke install` / `yoke refresh`.
- `keep_internal_only` — implementation detail behind the API boundary; agents structurally blocked from direct call.
- `keep_as_operator_debug` — command-shaped CLI/Python retained for operator use only; explicitly not the agent path.
- `delete` — purge from live code, docs, skills, packets, tests.

Dispositions are committed answers, not hedges. Sequencing — "decided, but not converting this family in the next 1–2 slices" — is expressed by follow-up roster position (AC-13/AC-18), not by a per-row flag. Genuine open questions go to the open-question registry (AC-16). Production telemetry and operator review catch wrong dispositions post-hoc.

### Five signals (informational columns)

1. **Authority location** — `yoke_core` / `project_repo` / `external_system` / `none`. Where canonical truth lives.
2. **Execution constraint** — `must_run_yoke_side` / `must_run_project_side` / `agent_choice`. Hard physical or capability constraint.
3. **Frequency** — `very_high_per_min` / `regular_per_session` / `rare_per_phase_or_less`. Derived from `runtime_event_summary` where possible.
4. **Agent token cost** — `agent_invisible` / `small_under_1k_tokens` / `large_over_1k_tokens`. LLM context consumed when the agent reads the response. Distinct from network payload.
5. **Current shape** — `shell_recipe` / `python_module` / `db_router` / `service_client` / `registered_function` / `http_route` / `direct_import` / `none`. Today's implementation surface.

### What these columns are NOT

These columns are not architectural location labels. The Gen 3 architectural question ("does this live in cloud, in a project-local package, or run locally via harness tools?") is answered by the combination of signals #1 and #2 plus the disposition — not by a single bucket. Authority note: user/auth identity and API-token scope are the future permission boundary. Machine/environment information may be stored/displayed as diagnostic metadata, but this ticket must not introduce machine-level permissions.

## Per-Config Classification

Applied at the **config family** level (expected ~5-10 families — to be discovered by the config-inventory runner; candidates include `lint_modes`, `session_lifecycle_windows`, `hook_defaults`, `recovery_windows`, `doctor_settings`, `harness_manifest`, `project_capabilities`, `schema_defaults`).

Each config family carries one disposition:

- **`env_side`** — the config value lives in each cloud Yoke env's runtime state (e.g. a `runtime_config` table in each env's Postgres, or each env's `data/config` filesystem when running locally). Agents fetch live values via a Yoke function call (typically `config.get` or per-family equivalent). The install bundle does NOT bake env-side config values into static files; the bundle teaches the agent how to fetch them.
- **`project_side`** — the config value lives in the project repo, git-tracked per branch, in an explicit project file. Examples: env binding (`data/connected-env.json`), project identity (`.yoke/project.json`), version pinning per env. Operator manages directly via the project repo.
- **`delete`** — the config no longer exists in Gen 3. The runtime behavior it controlled is either removed, hardcoded to one value, or absorbed by a different config family.

Each config family carries: `affected_surfaces` (which surface families read this config so the family pages can cross-reference), `served_via_function_id` (when `env_side`, names the Yoke function that returns the value), `project_file_location` (when `project_side`, names the project file and field schema), `deletion_rationale` (when `delete`), and a one-paragraph `rationale`.

## Per-Family Design Constraints From Live Friction

The first live agent edit against this ticket surfaced four design constraints that the family-disposition pages MUST resolve when classifying the `items` family (and analogous write families: `claims`, `lifecycle`, `workflow_item`, etc.). Each is a small API-shape decision but each cost real tokens when missing.

**Constraint 1 — Actor identity resolves server-side from session_id.** The `FunctionCallRequest.actor` model currently requires both `session_id` AND `actor_id`. Agents look up `actor_id` via `SELECT actor_id FROM harness_sessions WHERE session_id=...` — one round trip per call. The design page for any `wrap_as_function` family with claim verification MUST name whether the function-call surface (`POST /v1/functions/call` and the in-process `dispatch()`) accepts `{"session_id": ...}` alone and resolves `actor_id` server-side. Recommended decision: yes, resolve server-side; `actor_id` becomes optional on the request model. The handful of call sites that pass actor_id explicitly stay valid (server-side resolution wins when both are present and agree; mismatch returns the existing `actor_identity_gate` denial).

**Constraint 2 — Claim auto-acquire / auto-release for plan-stage spec writes.** Today every `items.structured_field.replace` call requires the calling session to hold a work claim on the item, even for `refined-idea` items where the operation is purely plan-stage authoring. Agents acquire-then-release on every edit (two extra round trips). The design page for `items.structured_field.replace` MUST decide whether the dispatcher auto-acquires-and-releases an item claim when the target is in a plan-stage status (`idea`, `refining-idea`, `refined-idea`, `plan-drafted`, `refining-plan`, `planned`) AND no other session holds the claim. Recommended decision: auto-acquire-and-release for plan-stage statuses only; implementation/review/polish statuses keep the explicit-claim contract (real concurrent-write risk).

**Constraint 3 — Canonical agent shape teaching, per family.** Every `wrap_as_function` family page MUST carry at least one concrete copy-pasteable invocation example for each transport (in-process CLI adapter, in-process Python dispatch, HTTP curl), not just the function-id reference. The teaching surfaces (packets, AGENTS.md, skill prose) carry generic principles today; agents reach for Python authoring as a fallback when no concrete recipe is visible for their specific operation. The HTTP route (`POST /v1/functions/call`) is live today via `python3 -m yoke_core.tools.api_server start` — Gen 3 work is auth and cloud transport, not the wire shape.

**Constraint 4 — Hooks family is composite-disposition.** Already named in `## Per-Surface Classification`; restated here so the design page is unambiguous: the `hooks.install_config` row (curl line in `.claude/settings.json` / `.codex/hooks.json`) gets `static_install_artifact`; the `hooks.policy_logic` row (server-side `/hooks/evaluate` dispatcher + per-event policy functions) gets `keep_internal_only`. The family page names both halves explicitly so the matching conversion slice does not re-discover the split.

These constraints emerged from one talmud session. The follow-up roster (deliverable #6) MUST surface analogous design questions for the next 1–2 conversion slices it recommends, so each slice's family page starts pre-loaded with API-shape decisions the analysis already encountered.

## Constraint 1 — landed
**Landed in YOK-1781.** The amendment shipped: `FunctionCallRequest.actor.actor_id` is now `Optional[str] = None`, and the dispatcher resolves it server-side from `harness_sessions` keyed on the bound `session_id`. A supplied `actor_id` that disagrees with the resolved value is rejected with `actor_id_mismatch` (mutating functions only; read-only callers pass through). The teaching sweep across AGENTS.md and the rendered agent packets is part of the same slice. Family pages downstream of this constraint can treat it as resolved.
