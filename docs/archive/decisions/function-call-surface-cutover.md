# Yoke-owned operations are first-class API calls

## Context

Every Yoke mutation — structured-field replace, additive transforms,
section upsert, Progress Log append, epic-task body replace, DB-claim
amend, work-claim acquire / release, path-claim register / widen,
QA-run record-verdict, board rebuild — used to flow through a thicket of
shell-only CLI surfaces with different argument shapes, different stdin
contracts, and no shared envelope. Operator-facing prose taught the CLI
verbs directly; new mutations meant new shell choreography. Agents
re-invented the same `mktemp` / heredoc / `--stdin` pipes per skill,
and there was no machine-readable inventory of "what can Yoke do?".

## Decision

All Yoke mutations are typed function calls dispatched through
`runtime.api.domain.yoke_function_dispatch.dispatch`. Each call carries
a typed `FunctionCallRequest`:

```jsonc
{
  "function": "...",
  "request_id": "<uuid>",
  "actor": {"session_id": "...", "actor_id": "..."},
  "target": {"kind": "item|epic_task|qa_requirement|...", "...": "..."},
  "payload": {"..."},
  "preconditions": {},
  "options": {}
}
```

Handlers live under `runtime/api/domain/handlers/`; registration files
group related operations (one per per-family slice). The dispatcher
emits `YokeFunctionCalled`, `DispatcherIdempotencyReplay`, and
`DispatcherDownstreamDegraded` events for observability.

The CLI surfaces (`db_router items update`, `item_field_transform`,
`epic task-update-body`, `service_client db-claim-amend`,
`service_client claim-work`, etc.) are **retained operator/debug
adapters** that build the matching `FunctionCallRequest` internally and
dispatch through the same registry. Agent prose, skill prose, and
dispatch context teach the function id; the CLI invocations remain valid
as operator/debug examples and are clearly labelled as such.

The canonical inventory lives at `docs/function-inventory.md`,
machine-generated by `runtime.api.domain.function_inventory_render` from
the live registry. It reports zero unclassified hits and zero
non-allowlisted function-covered recipe hits. Doctor's recipe-residue
check (`HC-recipe-residue`) catches drift; the
`runtime.api.test_zero_shell_proof` and `test_doc_hygiene` test suites
guard the inventory.

## Consequences

- Skills, agents, and dispatch context name function ids first.
- New mutations register under `runtime/api/domain/handlers/` and inherit
  the dispatcher's envelope, idempotency, downstream-degraded handling,
  and event emission for free.
- The "literal zero shell" contract holds: new launchers, hooks,
  helpers, installers, and test runners live behind `python3 -m
  runtime.api...` or packaged Python entrypoints, never tracked `.sh`.
- Recipe-residue guards (`HC-recipe-residue`, `test_zero_shell_proof`,
  `test_doc_hygiene`) keep the migration enforced.
