# Task 001: Schema extensions: new tables + ALTER TABLE epic_tasks

## Description
Extend the Yoke DB schema to support full epic lifecycle data. Add 6 new tables and 8 new columns to `epic_tasks`. All schema changes are made in `sprint-db.sh`'s `cmd_init()` function and must be idempotent.

Since SQLite lacks `ALTER TABLE ADD COLUMN IF NOT EXISTS`, create a helper function `_add_column_if_not_exists()` that uses `PRAGMA table_info` to check column existence before attempting the ALTER.

## Acceptance Criteria
- [ ] AC-1: `epic_task_history` table exists with columns: id, epic_slug, task_num, timestamp, from_status, to_status, note. Index on (epic_slug, task_num).
- [ ] AC-2: `epic_dispatch_chains` table exists with columns per FR-3. Unique constraint on (epic_slug, worktree).
- [ ] AC-3: `epic_reviews` table exists with columns per FR-4. Index on (epic_slug, task_num).
- [ ] AC-4: `epic_progress_notes` table exists with columns per FR-5. Unique constraint on (epic_slug, task_num, note_num).
- [ ] AC-5: `epic_simulations` table exists with columns per FR-6. Unique constraint on (epic_slug, phase).
- [ ] AC-6: `epic_tasks` has new columns: body, github_issue, branch, worktree_path, blocked_by, max_attempts (DEFAULT 5), agent_id, last_heartbeat.
- [ ] AC-7: Running `sprint-db.sh init` twice in a row produces no errors (idempotent).
- [ ] AC-8: The 7 existing `epic_tasks` rows are preserved after schema migration (no data loss on ALTER TABLE).

## Test Plan
- Unit tests: Run `sprint-db.sh init` on a fresh DB, verify all tables exist via `PRAGMA table_info`. Run init again, verify no errors. Verify existing epic_tasks data preserved.
- Manual verification: Run init against production DB, check `PRAGMA table_info(epic_tasks)` shows all new columns.
