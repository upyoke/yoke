# Browser Scenario Schema

Defines the structured JSON format for `qa_requirements.success_policy` when `qa_kind` is `browser_smoke` or `browser_diff`. This schema is the contract between:

- **Seeding logic** (shepherd step 5m, advance SKILL.md) -- generates skeleton scenarios
- **Tester agent** -- refines skeletons into executable steps
- **`yoke_core.domain.browser_client exec`** -- executes steps against the browser substrate via `step-executor.js`

**Vocabulary alignment:** This schema uses the **executor vocabulary** from `step-executor.js` as the canonical reference. Field names (`route`, `target`, `check`) and action names (`navigate`, `assert`, `wait_for`, `screenshot`) match what the executor accepts directly. No translation layer is needed between this schema and the executor.

**Canonical vocabulary only:** `step-executor.js` rejects non-canonical scenario fields such as `url`, `selector`, and `wait`. Stored scenario JSON uses the executor vocabulary directly.

## Schema Definition

```json
{
 "type": "browser_scenario",
 "base_url": "string -- project base URL (e.g., https://yok-941.buzzabuzz.com)",
 "steps": [
 {
 "action": "navigate|click|type|fill_form|assert|screenshot|wait_for|delay|scroll|hover|select",
 "route": "string -- target URL for navigate action (relative to base_url or absolute)",
 "target": "string -- CSS selector, accessibility selector, or ref (e.g., ref:7)",
 "value": "string -- input value for type, expected text for assert (optional)",
 "check": "string -- assertion check type for assert action (visible|hidden|text_contains|text_equals|count_gte|count_eq)",
 "expected": "string|number -- expected value for assert checks (text_contains, text_equals, count_eq)",
 "min_count": "number -- minimum count for count_gte assert check",
 "fields": "object -- key-value pairs of target:value for fill_form action",
 "capture": "boolean -- REQUIRED true on screenshot steps to actually save the image",
 "fullPage": "boolean -- capture full page screenshot (default false)",
 "duration": "number -- delay duration in milliseconds for delay action (default 1000)",
 "duration_ms": "number -- alias for duration (delay action)",
 "timeout_ms": "number -- per-step timeout in milliseconds (default 5000)",
 "refined": "boolean -- false = skeleton placeholder, true = Tester-confirmed",
 "source_ac": "string -- AC-N reference linking this step to an acceptance criterion (optional)",
 "semantic_keyword": "boolean -- true when this assert was generated by the semantic keyword fallback (optional)"
 }
 ]
}
```

## Action Reference

| Action | Required Fields | Description |
|--------|----------------|-------------|
| `navigate` | `route` | Navigate to URL (relative to `base_url` or absolute) |
| `click` | `target` | Click an element |
| `type` | `target`, `value` | Type text into an input element (keyboard input with optional `delay`) |
| `fill_form` | `fields` | Fill multiple form fields; `fields` is `{"target": "value", ...}` |
| `assert` | `target`, `check` | Assert a condition on an element (see Assertion Checks below) |
| `screenshot` | `capture: true` | Capture a screenshot. **Must include `"capture": true`** or no image is saved. |
| `wait_for` | `target` | Wait for the target element to be visible within timeout |
| `delay` | (optional `duration`) | Pure time delay -- waits `duration` ms (default 1000). No DOM target needed. |
| `scroll` | (optional `target`, `x`, `y`) | Scroll to element or by pixel offset |
| `hover` | `target` | Hover over an element |
| `select` | `target`, `value` | Select an option from a dropdown |

## Assertion Checks (for `assert` action)

The `assert` action uses the `check` field to determine what to verify:

| Check | Required Fields | Description |
|-------|----------------|-------------|
| `visible` | `target` | Verify element is visible in the DOM |
| `hidden` | `target` | Verify element is hidden |
| `text_contains` | `target`, `expected` | Verify element text contains expected string |
| `text_equals` | `target`, `expected` | Verify element text equals expected string (trimmed) |
| `count_gte` | `target`, `min_count` | Verify at least N matching elements exist |
| `count_eq` | `target`, `expected` | Verify exactly N matching elements exist |

## Skeleton vs Refined

Scenarios are generated in two phases:

1. **Skeleton (seeding time)**: Generated by shepherd/advance when a browser-testable item is classified. Steps have `"refined": false` and use placeholder selectors derived from AC text keywords. The skeleton is a starting point -- it is executable (all actions are valid executor vocabulary) but assertions use placeholder selectors.

2. **Refined (validation time)**: The Tester agent reads the skeleton from `success_policy`, explores the codebase to find real selectors, and replaces placeholders. All steps must have `"refined": true` before the scenario is considered fully validated.

### Skeleton Generation Rules

Browser-QA metadata is an authoritative structured field on every item, not an after-the-fact inference. `/yoke idea` writes a validated `browser_qa_metadata` object at creation time (keys: `browser_testable`, `visual_outcome`, `browser_routes`, `browser_timing_hints_ms`), `/yoke refine` can correct it through the sanctioned structured-field write surface, and the seeding path reads it directly. Non-browser items hold the explicit negative-default object, so the generator produces zero scenarios for them.

The seeding logic generates skeletons by:
- Reading the validated metadata via `python3 -m yoke_core.cli.db_router items get YOK-N browser_qa_metadata` (or, from Python, `yoke_core.domain.qa_requirements.read_browser_qa_metadata`). If `browser_testable=false`, no browser scenarios are emitted.
- **Multi-route iteration:** One `browser_smoke` requirement per entry in `browser_routes`. An empty route list defaults to a single scenario against `"/"`.
- Looking up the project's ephemeral URL from `ephemeral_environments` (fallback: project capability `base_url`, then `http://localhost:3000`).
- Generating per route: `navigate` step, `delay` step (see settle-delay floor below), and `screenshot` step with `capture: true`. Timing splits emit one additional `browser_smoke` per `browser_timing_hints_ms` entry so multi-capture strategies keep distinct success policies.
- **Pre-screenshot settle-delay floor (2000 ms):** Every seeded `navigate` → `screenshot` sequence gets at least a 2000 ms `delay` between the two. The floor keeps the first screenshot from capturing a spinner, skeleton loader, or un-hydrated font swap instead of the real render. An AC-derived timing hint **larger** than 2000 ms replaces the floor; it does not stack with it. A hint smaller than 2000 ms is raised to 2000 ms. `build_browser_scenario_policy()` in `packages/yoke-core/src/yoke_core/domain/qa_requirements.py` is the single source of truth for this rule; the sibling `min_delay_before_first_screenshot(timing_hint_ms)` helper exposes the same `max(2000, hint)` math for callers that need the number in isolation.
- **Timed captures:** For each entry in `browser_timing_hints_ms`, an additional scenario is emitted whose `delay` step uses that exact hint (subject to the floor rule above).
- Marking all steps with `"refined": false`. Richer per-step assertions (selectors, quoted text, semantic keywords) are introduced during refine or polish against the already-seeded requirement, not speculated from prose at seeding time.

### Tester Refinement Rules

The Tester agent:
- Reads the skeleton from `success_policy`
- Replaces placeholder selectors with real CSS/accessibility selectors from codebase exploration
- Adds interaction steps (click, type, wait_for) as needed
- Sets `"refined": true` on each completed step
- Persists the refined scenario back to `qa_requirements.success_policy`
- All steps must be `"refined": true` before the scenario is considered fully validated

## Mapping to `yoke_core.domain.browser_client`

Each step in the `steps` array maps directly to a `browser_client exec step` call:

```sh
# Execute a single step
python3 -m yoke_core.domain.browser_client exec step '{"action": "navigate", "route": "/login"}' --base-url https://yok-941.buzzabuzz.com
python3 -m yoke_core.domain.browser_client exec step '{"action": "assert", "target": ".login-form", "check": "visible"}' --base-url https://yok-941.buzzabuzz.com
python3 -m yoke_core.domain.browser_client exec step '{"action": "screenshot", "capture": true}' --base-url https://yok-941.buzzabuzz.com
```

The `browser_client exec step` command accepts the same JSON fields as this schema.

## browser_smoke vs browser_diff

| Requirement Kind | Scenario Type | Required Artifacts | When Seeded |
|-----------------|---------------|-------------------|-------------|
| `browser_smoke` | Navigate + assert + screenshot | `screenshot` | `browser_testable=true` |
| `browser_diff` | Screenshot + baseline comparison | `diff_image` | `visual_outcome=true` |

Items with `visual_outcome=true` receive a `browser_smoke` requirement per route. A `browser_diff` requirement is only seeded when a baseline screenshot exists for the same route — without a baseline, the diff is meaningless. The diff threshold defaults to 5% and is configurable per-project via `project_capabilities` (`browser-qa.diff_threshold`).

**Multi-route seeding:** When `browser_qa_metadata` lists multiple `browser_routes`, `build_browser_requirements_from_metadata` emits one `browser_smoke` requirement per route, plus one additional `browser_smoke` per entry in `browser_timing_hints_ms`. A `browser_diff` per route is added when `visual_outcome=true` and `include_diff=true` (callers gate this flag on a baseline-exists check). Per-route granularity keeps pass/fail signal scoped to the surface each scenario actually touches.

## Screenshot Evaluation Gate

The mechanical orchestrator (`yoke_core.domain.browser_qa`) runs scenario steps and checks assertion results, but it cannot judge whether captured screenshots are visually correct. For example, `text_contains "Login"` passes on any login page regardless of the intended theme, and mechanical assertions cannot detect rendering anomalies like raw Unicode escape sequences displayed as literal text.

After the orchestrator captures screenshots, the row lands with `execution_status='captured'` and `verdict=NULL` — no quality judgment has been made yet. The **browser-QA gate** in `advance/browser-qa.md` (screenshot evaluation section) then performs the inspection with **two independent checks**:

1. Collects screenshot artifact paths and the capture `qa_run_id` from the orchestrator's stdout JSON
2. Reads each screenshot image
3. **Visible-defect scan:** Inspects for any rendering anomaly a human would immediately flag as broken — raw escaped text, broken images, overlapping elements, clipped content, missing fonts, layout collapse, blank sections, console error overlays, or garbled text. This check is independent of the ACs; a page can satisfy every AC and still be visibly broken
4. **AC consistency check:** Evaluates whether the screenshot content matches the item's acceptance criteria
5. Both checks must pass. Inspection outcome is written with `yoke qa run complete --requirement-id <id> --run-id <capture_run_id> --verdict pass|fail` — the same row as the capture, preserving one-row-per-capture audit. Only then does `yoke qa screenshot-evidence satisfy --item YOK-N` bridge the capture into `ac_verification` (its guard rejects capture-only rows with exit 3).

This ensures the pipeline never reports `verdict='pass'` from infrastructure success alone — the verdict always reflects inspection outcome.

## Executor Types

| Executor Type | Description | Gate Enforcement |
|--------------|-------------|-----------------|
| `agent` | Self-reported by Tester agent (no substrate execution) | **Rejected** for `browser_smoke`/`browser_diff` at write time (`python3 -m yoke_core.cli.db_router qa run-add`) |
| `browser_substrate` | Executed via `yoke_core.domain.browser_client exec` with real browser | Allowed -- produces real artifacts |

## Rejected Aliases

The executor rejects any step containing `url` instead of `route`, `selector` instead of `target`, or `wait` instead of `delay` / `wait_for`. Stored `qa_requirements.success_policy` rows use canonical vocabulary.

## Example: Multi-Route Skeleton

An item with ACs spanning `/login` and `/forgot-password` generates **two** `browser_smoke` requirements:

**Requirement 1 — `/login` route:**
```json
{
 "type": "browser_scenario",
 "base_url": "https://yok-1018.buzzabuzz.com",
 "steps": [
 {"action": "navigate", "route": "/login", "refined": false, "source_ac": "setup"},
 {"action": "assert", "target": "body", "check": "text_contains", "expected": "yankees", "refined": false, "source_ac": "AC-1", "semantic_keyword": true},
 {"action": "delay", "duration": 12000, "refined": false, "source_ac": "timing"},
 {"action": "screenshot", "capture": true, "refined": false}
 ]
}
```

**Requirement 2 — `/forgot-password` route:**
```json
{
 "type": "browser_scenario",
 "base_url": "https://yok-1018.buzzabuzz.com",
 "steps": [
 {"action": "navigate", "route": "/forgot-password", "refined": false, "source_ac": "setup"},
 {"action": "assert", "target": "body", "check": "text_contains", "expected": "yankees", "refined": false, "source_ac": "AC-1", "semantic_keyword": true},
 {"action": "screenshot", "capture": true, "refined": false}
 ]
}
```

After Tester refinement:

```json
{
 "type": "browser_scenario",
 "base_url": "https://yok-1018.buzzabuzz.com",
 "steps": [
 {"action": "navigate", "route": "/login", "refined": true, "source_ac": "AC-1"},
 {"action": "assert", "target": "[data-testid='login-form']", "check": "visible", "refined": true, "source_ac": "AC-1"},
 {"action": "assert", "target": "h1.page-title", "check": "text_contains", "expected": "Welcome", "refined": true, "source_ac": "AC-2"},
 {"action": "screenshot", "capture": true, "refined": true}
 ]
}
```
